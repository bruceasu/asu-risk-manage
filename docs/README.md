1. 系统目标
系统核心目标有四个：
    1. 识别异常账户
从账户、交易、设备、登录、关系网络等数据中，发现偏离正常群体的账户。
    2. 识别高风险欺诈模式
不只找“离群点”，还要识别团伙、撞库、羊毛党、洗钱链路、盗号、套利等模式。
    3. 输出可解释结论
不只是打分，还要说明“为什么可疑”。
    4. 支持人工处置闭环
输出调查报告、审查队列、证据链，并把人工结论回流给模型。

2. 总体架构
建议采用“规则 + 统计 + 图谱 + 向量 + AI解释”的组合架构，而不是只靠单一模型。
2.1 逻辑分层
A. 数据接入层
输入数据包括：
    • 账户主数据：注册时间、实名状态、KYC等级、国家/地区、渠道来源
    • 登录行为：时间、IP、设备指纹、UA、地理位置、失败次数
    • 交易行为：金额、频率、币种、对手方、支付方式、充提记录
    • 设备与网络：设备ID、cookie、IP段、代理/VPN、ASN
    • 用户操作事件：修改手机号、改密码、绑定银行卡、提额、提现
    • 关系数据：账户-设备、账户-IP、账户-收款卡、账户-地址、账户-邀请关系
    • 标签数据：历史封禁、投诉、人工审核结论、黑名单命中
    • 外部数据：制裁名单、黑产IP库、代理IP库、高风险国家/地区等
B. 特征与画像层
形成三类特征：
    • 规则特征：是否命中已知风险规则
    • 统计特征：行为聚合和时序特征
    • 向量特征：将账户行为压缩成 embedding / representation
C. 风险检测层
并行运行多个检测器：
    • 规则引擎
    • 监督模型（有标签时）
    • 无监督异常检测
    • 图谱团伙检测
    • 时序/突变检测
    • 向量相似群组检测
D. 决策编排层
将多个模型与规则结果融合成统一风险评分：
    • account_risk_score
    • case_priority
    • risk_reasons
    • recommended_action
E. 调查与报告层
输出：
    • 异常账户列表
    • 风险原因摘要
    • 关联账户图谱
    • 自动调查报告
    • 人工审核建议
F. 反馈学习层
将人工审核结论写回：
    • confirmed_fraud
    • false_positive
    • watchlist
    • cleared
用于后续规则优化、模型再训练、阈值调整。

3. 向量特征设计
你特别提到“向量数据”，这里我把向量设计分成三层：显式特征向量、时序行为向量、关系图谱向量。生产中通常三者并存。

3.1 显式账户特征向量
这是最基础、最稳定的一层。每个账户在固定时间窗口内生成一条向量。
时间窗口建议
建议至少做三套窗口：
    • 1天窗口：抓短期攻击、盗号、突发套利
    • 7天窗口：抓中短期异常活跃
    • 30天窗口：抓长期模式偏差
特征类别
A. 账户基础属性
    • 账户年龄
    • KYC等级
    • 是否实名
    • 注册渠道
    • 国家/地区
    • 注册设备数量
    • 注册IP风险等级
    • 账户类型（个人/企业/代理）
B. 登录行为特征
    • 近1/7/30天登录次数
    • 登录失败率
    • 首次登录到首次交易时长
    • 不同国家登录次数
    • 夜间登录占比
    • 异常时段活跃度
    • 设备切换频率
    • IP切换频率
    • 高风险IP占比
    • VPN/代理命中率
    • 异地跳跃登录次数
C. 交易行为特征
    • 交易笔数
    • 总交易额
    • 平均交易额
    • 交易额标准差
    • 最大/最小交易额
    • 小额高频交易占比
    • 大额突增比率
    • 交易时间分布熵
    • 充值后快速提现比例
    • 入金/出金比
    • 失败交易率
    • 对手方数量
    • 新对手方占比
    • 单一对手集中度
    • 可疑国家/地区交易占比
D. 敏感操作特征
    • 改密码次数
    • 改手机号次数
    • 改绑定卡次数
    • 设备重绑次数
    • 风险操作后短时交易/提现次数
E. 关系与共享特征
    • 共用设备账户数
    • 共用IP账户数
    • 共用银行卡账户数
    • 共用收款地址账户数
    • 邻居账户中高风险账户占比
F. 标签与历史特征
    • 历史告警次数
    • 被投诉次数
    • 曾经人工审核次数
    • 历史误报概率
    • 黑名单关联深度

3.2 时序行为向量
显式聚合特征能抓很多问题，但对行为节奏不够敏感。建议加时序向量。
设计方式
将账户在时间轴上的行为编码成序列，例如：
    • 每小时交易次数序列
    • 每小时交易金额序列
    • 每日登录次数序列
    • 每日设备切换序列
    • 敏感操作事件序列
可进一步提炼成：
    • 周期性
    • 突增程度
    • 稳定性
    • 熵
    • burstiness（爆发性）
    • 自相关指标
可抓住的欺诈现象
    • 某一时间段突然密集活动
    • “养号后集中收割”
    • 登录-改密-提现的固定攻击链
    • 非人类操作节奏
    • 团伙批量脚本化行为

3.3 图谱/关系向量
欺诈通常不是单点问题，而是团伙问题。所以必须有关系图谱。
图节点
    • 账户
    • 设备
    • IP
    • 手机号
    • 银行卡
    • 地址
    • 收款账户
    • 邀请码
    • 浏览器指纹
    • 商户/对手方
图边
    • 账户使用设备
    • 账户登录IP
    • 账户绑定手机号
    • 账户关联银行卡
    • 账户与账户交易
    • 账户邀请账户
    • 账户共享设备/IP/卡
图特征
    • 度数
    • 聚类系数
    • PageRank
    • 连接的高风险节点比例
    • 团伙社区大小
    • 二跳/三跳黑名单接近度
    • 星型/链式/环形结构模式
    • 短时间内新建高关联团簇
图向量
可以将账户在图中的上下文编码成 embedding，用于：
    • 找相似团伙成员
    • 找与已知欺诈团伙“结构相似”的账户
    • 发现共享基础设施的异常群体
3.4 向量设计原则
不要只用原始 embedding
若你现有“账户向量”是某个模型直接生成的 embedding，可以用，但不要只依赖它。最好同时保留：
    • 原始可解释特征
    • 聚合统计特征
    • 图谱特征
    • embedding 特征
因为生产环境里，可解释性和稳定性与效果同样重要。
分层标准化
不同特征尺度差异很大，建议：
    • 金额类：log 变换
    • 计数类：winsorize + 标准化
    • 比例类：直接保留
    • embedding 类：L2 normalize
分群建模
不要拿所有账户用同一个分布建模。最好按业务类型分层：
    • 新户 / 老户
    • 个人 / 企业
    • 普通用户 / 高频交易用户
    • 国内 / 跨境用户
    • 低活跃 / 高活跃用户
否则异常检测会把“正常高活跃用户”误判成异常。

4. 异常检测模型设计
生产上建议采用“多检测器并行 + 融合评分”。

4.1 第一层：规则引擎
这是必须有的，不可替代。
作用
    • 捕捉已知明确风险
    • 给出强可解释告警
    • 补足模型盲区
    • 满足合规和运营要求
规则类型
静态规则
    • 同一设备注册多个账户
    • 高风险IP登录
    • 同一银行卡绑定多个账户
    • 充值后短期内快速提现
    • 短时间修改密码后提现
组合规则
    • 异地登录 + 新设备 + 大额提现
    • 多账户共享设备 + 同向交易 + 集中提现
    • 新户 + 夜间高频交易 + 对手高度集中
时序规则
    • 30分钟内连续失败登录后成功
    • 1小时内新增多个收款对象
    • 24小时内多次设备切换
输出
    • rule_hits
    • rule_severity
    • evidence

4.2 第二层：无监督异常检测
当标签不足时，这是核心。
可选模型
Isolation Forest
适合：
    • 大规模
    • 特征维数中等
    • 先做账户级异常筛查
优点：
    • 稳定
    • 易部署
    • 适合第一版上线
Local Outlier Factor
适合：
    • 找局部偏离点
    • 某个细分群体内异常
缺点：
    • 大规模性能较差
    • 在线部署不如 IF 简单
One-Class SVM
适合：
    • 中小规模
    • 分布相对清晰
缺点：
    • 大规模不友好
Autoencoder / Variational Autoencoder
适合：
    • 高维复杂行为
    • 需要学习正常行为重建误差
优点：
    • 对复杂非线性模式更强
缺点：
    • 调参与稳定性要求更高
实际建议
第一阶段先上：
    • Isolation Forest
    • LOF（针对高风险子群）
    • Autoencoder（后续增强）

4.3 第三层：监督学习模型
当你已经有一定量人工审核标签时，应该加监督模型。
目标标签
    • fraud_confirmed
    • account_takeover
    • mule_account
    • bonus_abuse
    • card_fraud
    • false_positive
常用模型
    • XGBoost / LightGBM
    • CatBoost
    • Logistic Regression（做基线和解释）
优势
    • 可直接学习哪些特征更接近“确认欺诈”
    • 能输出更接近业务目标的排序分数
注意
监督模型会受标签偏差影响，因为被人工看过的账户不代表全量分布，所以仍需保留无监督层。

4.4 第四层：图谱风险检测
这是团伙欺诈识别的关键。
典型检测任务
    • 社区发现：找设备/IP/卡高度共享的群体
    • 风险传播：已知黑名单附近节点加权升分
    • 团伙模板匹配：识别星型团伙、漏斗转移、链式洗钱
典型场景
    • 同一设备控制多个账户
    • 多个账户资金向同一收口账户集中
    • 多账户在相近时段从同一网段登录
    • 新注册账户快速形成密集交易网络
输出
    • graph_risk_score
    • suspicious_cluster_id
    • cluster_size
    • linked_bad_accounts

4.5 第五层：相似账户检索
如果你已有一批已确认欺诈账户，可以用向量相似检索。
逻辑
    • 为已确认欺诈账户构建“高风险原型集”
    • 对新账户做向量近邻搜索
    • 找与高风险原型相似的账户
适合发现
    • 同类型套路账户
    • 行为模板复用
    • 批量注册、批量套利模式
注意
相似不等于欺诈，所以只能作为辅助信号。

5. 风险评分融合设计
不要直接使用单一模型分数。建议做一个风险编排器。
5.1 融合输入
    • rule_score
    • anomaly_score_unsupervised
    • supervised_risk_score
    • graph_risk_score
    • similarity_score
    • temporal_burst_score
    • historical_alert_score
5.2 融合方式
建议三阶段：
阶段1：标准化
将各子模型输出统一映射为 0–100。
阶段2：分层加权
按账户类型、地区、业务线设置不同权重。
例如：
    • 新户：规则和图谱权重更高
    • 老户：行为异常和时序突变权重更高
    • 企业户：交易网络和集中度权重更高
阶段3：策略阈值
输出三档：
    • 低风险：仅观察
    • 中风险：进入人工审核
    • 高风险：触发限制动作或优先审查

6. 自动发现异常账户流程
这里给出完整发现链路。
6.1 日常批处理发现
每日/每小时跑一次：
    1. 更新窗口特征
    2. 生成账户向量
    3. 跑规则引擎
    4. 跑无监督异常检测
    5. 跑图谱关联分析
    6. 融合风险评分
    7. 输出高风险候选账户列表
输出字段建议包括：
    • account_id
    • final_risk_score
    • risk_level
    • top_reasons
    • rule_hits
    • anomaly_rank
    • graph_cluster_id
    • related_accounts_count
    • last_activity_time
    • recommended_action
6.2 实时发现
对关键事件做实时打分：
    • 登录
    • 改密
    • 换绑
    • 提现申请
    • 大额交易
    • 新设备登录
实时链路：
事件进入 → 更新轻量特征 → 实时规则判断 → 调用在线风险模型 → 输出动作
动作可以是：
    • 放行
    • 二次验证
    • 延迟处理
    • 人工复核
    • 临时限制
6.3 群体发现
每天额外跑一次“群体异常发现”：
    • 新增共享设备簇
    • 新增共享银行卡簇
    • 新增高密度交易社区
    • 新增相似行为群组
这一步很重要，因为团伙通常不只是单个账户异常。

7. 自动生成调查报告设计
这里 AI 的作用非常适合：不是替代检测，而是生成调查解释和报告。

7.1 报告目标
一份合格的调查报告至少要回答：
    1. 这个账户为什么被系统判为可疑？
    2. 证据是什么？
    3. 与哪些风险实体有关？
    4. 风险类型是什么？
    5. 建议怎么处置？

7.2 报告输入结构
建议先由系统把结构化证据整理好，再交给 AI 生成自然语言。不要让 AI 直接读原始海量日志。
输入应包括
A. 基本信息
    • 账户ID
    • 注册时间
    • KYC状态
    • 账户类型
    • 最近活动时间
B. 风险总览
    • 最终风险分
    • 风险级别
    • 风险类型候选
    • 是否命中高优先级规则
C. 关键证据
    • 命中规则列表
    • 主要异常特征 Top N
    • 与正常群体偏离的指标
    • 图谱关联摘要
    • 近7天关键事件时间线
D. 对比基线
    • 同群体平均交易频率
    • 同群体平均设备数
    • 同群体平均提现时延
    • 账户偏离倍数
E. 关联实体
    • 共享设备
    • 共享IP
    • 共享银行卡
    • 关联高风险账户数
    • 所属可疑团伙 ID
F. 建议动作
    • 建议人工审核
    • 建议限制提现
    • 建议补充KYC
    • 建议列入观察名单

7.3 报告模板
建议分四部分。
1) 执行摘要
用两三句话说明风险结论。
例如结构：
    • 账户在近24小时内表现出显著异常行为
    • 主要风险来自设备共享、快速资金转移和高风险IP登录
    • 与一个已有高风险账户群存在多重关联
2) 关键发现
分条写：
    • 行为异常
    • 关系异常
    • 交易异常
    • 历史异常
3) 证据明细
列出：
    • 时间线
    • 规则命中
    • 关键指标偏离
    • 图谱关系截图/描述
4) 处置建议
    • 风险等级
    • 建议动作
    • 是否升级人工复核
    • 后续监控项

7.4 AI生成报告的边界
AI只做：
    • 摘要
    • 解释
    • 整理证据
    • 生成调查语言
AI不应直接决定：
    • 是否封禁
    • 是否冻结资金
    • 是否认定违法
这些必须由规则、模型和人工流程共同决定。

8. 风险类型体系设计
为了让报告和处置更清晰，建议建立风险 taxonomy。
例如：
    • ATO：账户被盗
    • Mule：跑分/洗钱中转账户
    • Bonus Abuse：套利/薅羊毛
    • Synthetic Identity：虚假身份
    • Collusive Ring：团伙协同
    • Promotion Abuse：活动滥用
    • Payment Fraud：支付欺诈
    • Device Farm：设备农场
    • Bot Automation：脚本批量操作
每个账户不只一个总分，还可以有：
    • overall_risk_score
    • risk_type_scores
这样报告能说清是“什么类型的可疑”。

9. 处置策略设计
检测只是前半段，后半段必须设计动作。
9.1 按风险级别分流
低风险
    • 进入观察名单
    • 加强监控
    • 不立即触发人工审核
中风险
    • 进入人工审核队列
    • 增加认证要求
    • 限制部分高风险动作
高风险
    • 实时拦截高风险操作
    • 延迟提现/转账
    • 冻结部分能力
    • 升级专员复核
9.2 动作与证据绑定
动作必须能解释：
    • 为什么拦截
    • 依据是什么
    • 是否可复核
    • 是否可申诉

10. 评估指标设计
不能只看 AUC。反欺诈更看业务效果。
10.1 模型指标
    • Precision@TopK
    • Recall
    • PR-AUC
    • False Positive Rate
    • 分群体稳定性
10.2 运营指标
    • 每日告警量
    • 人工审核通过率
    • 告警命中率
    • 高风险账户确认率
    • 平均调查时长
    • 每个告警的有效损失拦截额
10.3 策略指标
    • 被拦截欺诈金额
    • 误伤正常用户比例
    • 用户申诉率
    • 审核积压量
    • 从告警到确认的时间

11. 数据与工程实现建议
虽然你现在不要代码，但设计阶段最好先把工程边界定义清楚。
11.1 数据分层
    • ODS：原始事件日志
    • DWD：清洗后明细
    • DWS：聚合特征
    • ADS：风控宽表、告警表、案件表
11.2 核心表
建议至少有：
    • account_feature_snapshot
    • account_embedding_snapshot
    • account_risk_score_daily
    • rule_hit_log
    • suspicious_cluster_table
    • investigation_case_table
    • reviewer_feedback_table
11.3 运行方式
    • 批处理：小时级、天级
    • 流处理：关键事件实时打分
    • 图分析：天级或小时级
    • AI报告：对中高风险账户异步生成

12. 上线路线图
不要一开始就做得很重。建议分阶段。
Phase 1：可用版本
目标：快速上线发现异常账户
包括：
    • 基础账户特征
    • 规则引擎
    • Isolation Forest
    • 简单风险融合
    • 人工审核列表
    • 固定模板报告
Phase 2：增强版本
目标：减少漏报，提高解释能力
增加：
    • 图谱特征
    • 团伙检测
    • 风险类型分类
    • AI生成调查摘要
    • 人工反馈回流
Phase 3：成熟版本
目标：形成闭环
增加：
    • 实时评分
    • 监督模型
    • 图向量/行为向量
    • 策略编排
    • 主动学习和阈值自适应
    • 案件管理系统联动

13. 一套推荐的最终方案
如果你要一个现实且稳健的组合，我建议是：
检测层组合
    • 规则引擎：抓已知风险
    • Isolation Forest：抓全局异常
    • LOF：抓局部异常
    • 图谱团伙分析：抓关联欺诈
    • 相似账户检索：抓模板化复用
    • 监督模型：有标签后做风险排序
特征层组合
    • 显式统计特征
    • 时序行为特征
    • 图谱关系特征
    • 账户 embedding
输出层组合
    • 风险总分
    • 风险类型分
    • Top reason codes
    • 证据链
    • AI 调查报告
